## 목차

### 📌 이슈 트래커 개요
이슈 트래커는 프로젝트 진행 중 발생하는 작업, 버그, 개선 사항 등을  
체계적으로 관리하기 위한 도구입니다.

이슈 트래커의 종류는 매우 다양하며,  
그중 일부 도구는 개인 또는 소규모 팀보다는  
기업 환경에서 주로 사용됩니다.

---

### 🧩 이슈 관리의 목적
- 작업 단위의 명확한 정의
- 진행 상황의 가시성 확보
- 팀원 간 작업 상태 공유
- 변경 이력 및 의사결정 기록 유지

이슈 관리는 단순한 할 일 목록이 아니라,  
**프로젝트 흐름을 관리하는 핵심 수단**입니다.

---

### 🔄 애자일 협업 방식 개요

#### 스크럼(Scrum)
- 짧은 주기의 반복적인 개발 방식
- 유의미한 작업 진행 상황만 공유
- 불필요한 논의는 최소화

스크럼 회의는  
진행 상황 공유와 장애 요소 파악에 목적이 있습니다.

---

### ⚙️ 협업에서 중요한 요소
- 반복 작업의 자동화
- 팀 전체가 공유하는 명확한 기준 지표 존재

공통 지표는  
프로젝트의 현재 상태를 팀원 모두가 동일하게 인식하도록 돕습니다.

---

## 이슈 구성 요소 정리

### 🧾 기본 개념
- **스토리(Story)**  
  하나의 사용자 시나리오 단위 작업  
  예: 사용자가 아이디와 비밀번호를 입력하여 로그인을 완료함

- **에픽(Epic)**  
  여러 개의 스토리를 묶은 상위 단위  
  예: 로그인, 회원가입 관련 스토리 묶음

---

### 🧩 이슈 필드 설명
- **요약(제목)**  
  이슈의 핵심 내용을 간결하게 표현한 제목

- **컴포넌트(Component)**  
  작업 영역 또는 역할 구분  
  예: 프론트엔드, 백엔드, UX, 인프라  
  컴포넌트 리드를 지정하면 담당자가 자동 설정될 수 있음

- **보고자(Reporter)**  
  이슈를 생성한 사람

- **수정 버전(Fix Version)**  
  해당 이슈가 반영될 예정인 릴리스 버전

- **레이블(Label)**  
  이슈 분류를 위한 태그 역할  
  이슈 성격을 빠르게 구분하기 위한 용도

- **릴리스(Release)**  
  버전 단위 배포 관리  
  각 버전에 포함된 작업 내역을 확인 가능

- **상위 항목(Parent)**  
  해당 스토리가 속한 에픽 지정

- **스프린트(Sprint)**  
  이슈가 포함될 스프린트 단위  
  초기에는 백로그에 두고 이후 스프린트로 이동 가능

- **스토리 포인트(Story Point)**  
  작업 난이도 또는 상대적 작업량  
  절대적인 시간 개념이 아닌 팀 내부 기준

- **에픽 네임(Epic Name)**  
  스토리에서 상위 에픽을 식별하기 위한 이름

---

### 🤖 이슈 설명 자동화
이슈의 설명 영역은  
AI를 활용해 초안 형태로 자동 생성할 수 있으며,  
최종 내용은 사람이 검토하여 확정합니다.

---

## JQL (Jira Query Language)

### 📌 JQL 개요
JQL은 이슈를 검색하기 위한 쿼리 언어로,  
특정 조건에 맞는 이슈를 필터링하는 데 사용됩니다.

구조적으로 SQL과 유사하지만,  
이슈 검색에 특화된 문법을 사용합니다.

---

### 🔍 JQL의 목적
- 원하는 이슈를 빠르게 검색
- 상태, 담당자, 버전 등 조건 기반 조회
- 복잡한 이슈 목록을 체계적으로 관리

---

### 🧩 JQL 활용 예시 개념
- 특정 상태의 이슈 조회
- 특정 담당자가 맡은 이슈 검색
- 특정 스프린트 또는 버전에 포함된 이슈 필터링
- 여러 조건을 조합한 이슈 검색

JQL은  
대규모 프로젝트에서 이슈를 효율적으로 관리하기 위한 핵심 도구입니다.
